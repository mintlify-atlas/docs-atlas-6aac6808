title: "p2panda-auth — decentralised group management"
description: "Eventually-consistent group management with Pull/Read/Write/Manage permissions, DAG-based operation ordering, and a pluggable Resolver for concurrency."
---

`p2panda-auth` provides decentralised group management with fine-grained, per-member permissions. Groups are modified by publishing signed operations that form a Directed Acyclic Graph (DAG), allowing concurrent operations to be authored offline and later merged. The system resolves conflicting states automatically using a pluggable `Resolver` trait. Four access levels — `Pull`, `Read`, `Write`, and `Manage` — each act as a superset of the levels below them, and optional conditions allow domain-specific access boundaries such as path-based scoping.

## Installation

```toml Cargo.toml
[dependencies]
p2panda-auth = "0.6.1"
```

## Access levels

Access levels form a strict ordering: `Manage > Write > Read > Pull`. Each level is a superset of all lower levels.

```rust
use p2panda_auth::{Access, AccessLevel};

// Construct access values for assignment to group members.
let pull_access = Access::pull();
let read_access = Access::read();
let write_access = Access::write();
let manage_access = Access::manage();

// Compare access levels.
assert!(manage_access > write_access);
assert!(write_access > read_access);
assert!(read_access > pull_access);

// Check specific level.
assert!(manage_access.is_manage());
assert!(write_access.is_write());
```

| Level | Can sync | Can read | Can write | Can manage |
|-------|----------|----------|-----------|------------|
| `Pull` | Yes | No | No | No |
| `Read` | Yes | Yes | No | No |
| `Write` | Yes | Yes | Yes | No |
| `Manage` | Yes | Yes | Yes | Yes |

## Access conditions

Conditions can be attached to any access level to create domain-specific access boundaries. Any type implementing `PartialOrd` can serve as a condition, allowing expiry timestamps, path hierarchies, or any application-defined rule.

```rust

let timed_access = Access::write()
    .with_conditions(ExpiresAt(100));
```

## Group operations and the DAG

Group state is modified by publishing signed group operations. Each operation includes an `action` (add, remove, promote, demote), a `previous` field for causal ordering, and optional `dependencies` for application-defined relationships. All operations form a DAG.

## Concurrency resolution

Concurrent group modifications can produce conflicting states (for example, two managers removing each other simultaneously). The `Resolver` trait allows plugging in custom resolution logic.

```rust
use p2panda_auth::traits::Resolver;

// The Resolver trait determines which operations to invalidate
// when concurrent conflicts are detected.
pub trait Resolver<ID, OP, M, C>
where
    ID: IdentityHandle,
    OP: OperationId + Ord,
{
    type State;
    type Error: Debug;

    /// Returns true if a full state rebuild is needed.
    fn rebuild_required(state: &Self::State, msg: &M) -> Result<bool, Self::Error>;

    /// Process all operations and update internal state (apply invalidation filter).
    fn process(state: Self::State) -> Result<Self::State, Self::Error>;
}
```

### Default "strong removal" resolver

The built-in resolver follows these rules:

1. Removal or demotion of a manager invalidates any concurrent actions by that member.
2. Mutual removals (two managers removing each other concurrently) are both applied, but their other concurrent actions are invalidated.
3. Re-adds are allowed: if Alice removes Charlie and then re-adds them, Charlie is still a member but all concurrent actions are invalidated.
4. Invalidation is transitive: invalidating an operation also invalidates all operations that depend on it.
